Индивидуальное домашнее задание

Индивидуальное домашнее задание (ИДЗ) является одним из видов работ в рамках курса Internet-технологии (2-й семестр).
Оценивается от 9 до 14 баллов (если не указано ограничение максимальной оценки) на основании защиты. Если вам интересно задание, но за него недостаточно баллов - обращайтесь (перед выполнением), предлагайте варианты расширения функционала, выполняйте и получайте желаемую оценку.
При этом 9 баллов - минимальная оценка, будет автоматом выставлена при отработке всех лабораторных работ, но без выполнения собственно ИДЗ. Крайний срок сдачи ИДЗ - день выставления зачета, дедлайн лабораторных работ к ИДЗ никак не относится. Одним студентом выполняется только один из вариантов ИДЗ, но один и тот же вариант ИДЗ может выполняться множеством студентов независимо друг от друга.
Доступны следующие темы заданий, на ваш выбор:

PDO, учетные данные. Приложение для хранения регистрационной информации - логинов, почты, пароля. Доступ к приложению только после ввода логина и пароля, соответствующих установленным в базе (полноценной регистрации пользователя делать в этом задании не нужно, для этого отдельный вариант есть). Пользователь имеет возможность добавление наборов регистрационных данных (логин или почта, пароль, примечания), их просмотр, изменение, удаление. Использовать хеширование (при хранении пароля для доступа к приложению самого пользоватея) и шифрование (для хранения сохраняемых данных).

PDO, кабинет пользователя. Управление учетными записями пользователей - часто реализуемый элемент веб-приложений. Учетные данные - логин/почта, пароль. Роли администратора, зарегистрированного пользователя, неизвестного пользователя. Администратор может управлять учетными записями зарегистрированных пользователей. Учетная запись самого администратора есть в системе изначально, после входа админ может изменить свой логин и пароль. Зарегистрированный пользователь после входа имеет доступ к своим учетным данным. Неизвестный системе пользователь, при попытке входа, получает предложение обратиться к администратору или зарегистрироваться самому. Максимальная оценка - 13 баллов.

MongoDB, телефонная книга. Возможность регистрации пользователя и хранение его данных с произвольным набором полей. Основные возможности для авторизованного пользователя:

  • добавление новой записи с произвольным набором полей (например, допустима одна запись с ФИО и телефоном, другая - с ФИО, пояснением, местом работы, почтой и телефоном, третья - с названием и телефоном и т.п. Т.е. набора полей жестко заданого нет);
  • фильтрация вывода документов по заданному пользователем параметру;
  • редактирование и удаление выбранного документа.
Клиентская часть должна включать в себя элементы для взаимодействия с пользователем (не знающем джейсона и правил для формирования документов! Т.е. предлагать пользователю что-то наподобие "Введите пары ключ:значение через запятую" нельзя). Для реализации серверной части предлагается использовать PHP или NodeJS.

Smarty/Twig. Используя серверный шаблонизатор Smarty или Twig, внести изменения в код лабораторной работы, посвященной работе с PDO, разделив код представления (шаблон, в который подгружаются данные) и контроллера (рендер шаблона и привязка к нему данных). Максимальная оценка - 12 баллов.

AngularJS. Взять за основу исходный код лабораторных работ, посвященных работе с PDO и организации ассинхронного обмена с помощью технологии AJAX. Если вариант лабораторной по PDO предполагал реализацию запросов на изменение - можно убрать их, оставив только вывод пользователю результатов выборки из базы по указанным им параметрам (обычно это три запроса). Перевести бекенд (часть приложения, ответственную за обработку запросов и работу с БД) на подготовку всех ответов в формате JSON. Фронтенд (клиентская часть приложения, которая и формирует запросы к серверу) переделать под использование фреймворка AngularJS: сделать привязку полей для ввода и вывода информации на странице к модели, отправку AJAX-запросов реализовать с помощью сервиса $http. Могут помочь старые методические указания для лабораторной по AngularJS. Максимальная оценка - 12 баллов.
Аналогичная переделка, но используя не AngularJS, а средства Vue.js, ограничена максимальными 14 баллами.

GitHub API. Веб-приложение позволяет получить перечень репозиториев для указанного пользователем аккаунта. Для выбранного репозитория вывести данные о нем (название, описание, URL, даты создания и последнего обновления, используемые языки) и перечень коммитов (дата, хеш и сообщение). Максимальная оценка - 12 баллов.

WG API. Приложение, использующее публичный программный интерфейс игровых сервисов Wargaming для получения данных о достижениях игрока в World Of Tanks. Приложение позволяет указать пользователю никнейм требуемого игрока и запрашивает у WG только не-приватные данные, не требующие использования access token (т.е. используя только application_id, для чего в любом случае нужна регистрация в сервисе). Выбор отображаемых пользователю достижений на ваше усмотрение. Возможно расширение задания, включая большее число используемых методов публичного API. Реализация задания предполагает использование JavaScript/PHP/NodeJS, допускается реализация в виде serverless-приложения. Максимальная оценка - 13 баллов.

API CoinMarketCap. Приложение, использующее API веб-ресурса для отслеживания цен на криптовалюты CoinMarketCap. Позволяет пользователю выбрать интересующую валюту и получить данные о ней, такие как ссылка на официальный сайт, описание, логотип, а также данные по курсу выбранной криптовалюты и его изменении за последние 24 часа. Реализация возможна как в виде десктопного приложения (C# - WinForms или WPF), так и как веб-приложение (сервер на PHP или NodeJS, клиент - HTML/CSS/JavaScript). Максимальная оценка - 13 баллов.

GAS, чтение. Веб-приложение для получения данных от стороннего сервиса - таблицы GoogleSpreadsheet, содержащей данные о фамилиях и почтовых адресах. Пользователю доступен выпадающий список с фамилиями, сформированный в начале работы приложения на основе существующих в таблице (т.е. для получения всех возможных фамилий потребуется запрос). С помощью выпадающего списка пользователь может выбрать фамилию, после чего к таблице выполняется запрос для получения соответствующего почтового адреса. Для взаимодействия с GoogleSpreadsheet использовать Google Apps Script. Загружать сразу все данные таблицы (до запроса от пользователя адреса для конкретной фамилии) нельзя. Максимальная оценка - 12 баллов.

GAS, чтение и запись. Приложение, работающее на стороне клиента, позволяющее вносить данные в гугловскую таблицу. Пользователю доступен элемент для ввода и кнопка, по нажатию на которую осуществляется запрос к Google Apps Script, который добавляет новую строку в таблицу. Также в приложении выводится текущее содержимое таблицы. Максимальная оценка - 12 баллов.

Electron. Приложение, написанное на Electron, предназначенное для обработки табличных данных. Из произвольного csv-файла считываются данные и выводятся в виде таблицы. Расположение файла указывается пользователем, также предусмотреть возможность генерирования файла, заполняя его значениями по умолчанию, и возможность добавлять новые записи в файл. Проверять при считывании соответствие числа столбцов в разных строках, если различается - сообщать пользователю об ошибке.

WebSocket/SocketIO, доска для рисования. Веб-приложение с использованием canvas для отображения и SocketIO для передачи данных между пользователями. Один рисует, все подключенные видят и вносят свои правки. Холст для рисования с общим доступом в режиме реального времени. Максимальная оценка - 12 баллов.

WebSocket/SocketIO, мониторинг погоды. Клиентская часть предоставляет возможность выбора населенного пункта и частоты обновления данных, а также получает от сервера данные о погодных условиях (текущая температура, осадки, уровень давления, скорость ветра) и отображает их пользователю. Сервер имитирует работу - случайным образом генерирует данные о погодных условиях выбранного города с заданным интервалом и отдает их подключенным клиентам. Максимальная оценка - 12 баллов.

WebSocket/SocketIO, диспетчер такси. Серверная часть приложения имитирует изменение данных (случайная генерация с заданным интервалом) о текущем расположении (широта и долгота) машин таксопарка. Клиентская часть приложения позволяет отслеживать в режиме реального времени данные о расположении всех машин таксопарка, а также добавлять новые машины, указывая номер, марку, ФИО водителя. После добавления машины данные по ней также начинают генерироваться сервером и рассылаться всем подключенным клиентам. Максимальная оценка - 13 баллов.

WebSocket/SocketIO, игра в слова. Пользователи подключаются к серверу, могут создать свою игру или подключиться к существующей игре. Игрок, создавший игру, при наличии минимум одного партнера, может дать старт. Сервер следит за очередностью игроков, которые пишут слова таким образом, чтобы каждое новое начиналось на последнюю букву предыдущего и не было ранее использовано в игре. Игрок может отказаться от своей очереди, тем самым выбывая из игры. Последний оставшийся является победителем. Максимальная оценка - 13 баллов.

WebSocket/SocketIO, игра в города. Пользователи подключаются к серверу, могут создать свою игру или подключиться к существующей игре. Игрок, создавший игру, при наличии минимум одного партнера, может дать старт. Сервер следит за очередностью игроков, которые пишут названия городов таким образом, чтобы каждое новое начиналось на последнюю букву предыдущего и не было ранее использовано в игре. Игрок может отказаться от своей очереди, тем самым выбывая из игры. Последний оставшийся является победителем. Перечень возможных названий городов хранится в базе данных на сервере.

WebSocket/SocketIO, составление слов. Игра, составление слов из случайно сгененированного на сервере набора букв. Пользователи подключаются к серверу, могут создать свою игру (указав число букв и ограничение по времени) или подключиться к существующей игре. Игрок, создавший игру, при наличии партнера, может дать старт - после чего обоим участникам сервер рассылает набор букв и начинает отсчет времени до конца игры. Игроки составляют слова из предложенного набора букв, указывая их построчно в текстовом поле, и по истечению времени содержимое текстовых полей каждого пользователя отправляется на сервер. По завершению игры сервер выводит игрокам список слов, найденных обоими участниками, и отдельные списки уникальных для каждого участника слов, с указанием победителя. Проверки на корректность составленных слов нет.

WebSocket/SocketIO, Данетки. Данетки (игра в ситуации). Пользователи подключаются к серверу, могут создать свою игру или подключиться к существующей игре. При создании игры описывается некая ситуация с вопросом, ответ на который могут получить остальные игроки, задавая вопросы на "да" и "нет". Данные новой игры созраняются на сервере (в файле или БД) для последующего повторного использования. Игрок, создавший игру, при наличии минимум одного партнера, может дать старт, после чего отвечает всем участникам поочередно, имея возможность ответить только "да", "нет", "вопрос некорректен", "игра завершена".

MediaWiki. Внедрить в свой ресурс (произвольный) документацию на движке MediaWiki. Документация должна содержать разделы о назначении проекта, используемых технологиях и реализованном функционале. Максимальная оценка - 12 баллов.
